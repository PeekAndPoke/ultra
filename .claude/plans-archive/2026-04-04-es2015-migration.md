# Plan: Set ES2015 Target for All Kotlin JS Compilations

## Status: SOLVED (2026-04-04)

The ES2015 migration is working. The root cause of the blocker (Kotlin/JS IR static factory methods) was
identified and resolved with a JS function bridge in `PreactLLC`. ES2015 target is enabled and verified in four
real projects; all core tests pass.

**Historical context:** originally blocked by two issues — the `PreactLLC` static-factory problem (Blocker 1) and
the konva `@JsModule` `import { default }` syntax error (Blocker 2). Both are resolved. Konva was removed from the
repo entirely; pixijs replaced it with hand-written minimal externals.

## Context

All ~40 Kotlin/JS modules in this monorepo previously compiled with the default ES5 target. Kotlin generated
prototype-based "classes" instead of real ES2015 `class X extends Y` syntax. ES2015 classes are required for
AudioWorklets, WebComponents, and other browser APIs that check the prototype chain.

---

## Solution (2026-04-04)

### The `PreactLLC` bridge fix

The core insight: `new` on a plain JavaScript function, when that function returns an object, discards the
dummy `this` JS just created and uses the returned object instead. This lets us hand Preact a constructor
it can call with `new`, while we actually produce the instance via Kotlin's static factory.

**File:** `kraft/core/src/jsMain/kotlin/vdom/preact/PreactLLC.kt` (`getLowLevelComponentCtor`)

```kotlin
internal fun getLowLevelComponentCtor(cls: KClass<out KraftComponent<*>>): dynamic {
    return compClass2LLC.getOrPut(cls) {
        // One unique subclass per Kotlin component class — helps Preact diffing.
        class Impl(props: PreactLLCProps, context: dynamic) : PreactLLC(props, context)

        // ::Impl invokes the Kotlin constructor (ES5) or static factory (ES2015).
        // Either way it returns a fully initialized Impl instance.
        @Suppress("UNUSED_VARIABLE", "unused")
        val factory = ::Impl

        // The real prototype chain: Impl -> PreactLLC -> preact.Component.
        // Preact detects it as a class component via this chain.
        @Suppress("UNUSED_VARIABLE", "unused")
        val proto = (Impl::class.js).asDynamic().prototype

        // Plain JS function, always new-able.
        // When Preact does `new bridge(props, ctx)`:
        //   1. JS creates a dummy `this` with __proto__ = proto
        //   2. bridge() calls factory() -> Kotlin returns a fully initialized Impl
        //   3. Since bridge returns an object, `new` uses it and discards the dummy
        //   4. Preact gets the properly initialized instance
        js("(function() { var C = function(p,c) { return factory(p,c); }; C.prototype = proto; return C; })()")
    }
}
```

This replaces the previous `::Impl` + prototype hack. It is backward compatible with ES5 (where `::Impl` returns
a direct constructor invocation) and correct under ES2015 (where the JS bridge's `new` absorbs the factory's
return value).

### Webpack `fullySpecified: false` requirement

Strict ESM requires file extensions on imports. Our dynamic imports don't include them
(e.g. `js("import('prismjs/components/prism-kotlin')")`), so webpack rejects them under ES2015 without this
rule.

**Required in every application webpack config:**

```js
// webpack.config.d/webpack.js
config.module.rules.push({
    test: /\.m?js/,
    resolve: {fullySpecified: false},
});
```

Also set `config.output.chunkFilename = "[id].[contenthash].js"` for CDN-safe async chunks.

### Konva removal

`@JsModule("konva")` in ES2015 mode generates `import { default } from 'konva'` — invalid JS since `default` is
a reserved keyword. Konva was unused anyway, so it was deleted: from `settings.gradle`, from the `kraft/addons/`
module list (wrapped in the PeekAndPoke settings), and from example code. Pixi.js took its place as the
canvas/graphics addon, with hand-written minimal externals rather than generated ones (see
`.claude/plans/kraft-ecosystem-and-tailwind.md`).

### Projects where ES2015 is enabled and working

- `kraft:examples:hello-world`
- `kraft:examples:fomanticui`
- `kraft:examples:addons`
- `funktor-demo:adminapp`

All 329 core tests pass with the new bridge. All 18 addon tests pass. The bridge is ES5-compatible, so modules
without ES2015 target continue to work unchanged.

## Historical Blockers (resolved)

### Blocker 1: Kotlin/JS IR static factory methods — RESOLVED

When targeting ES2015, the Kotlin/JS IR compiler generates **static factory methods** instead of real constructors
for classes that extend external classes. This is a compiler design choice: ES2015 requires `super()` to be called
before `this` is accessible, and Kotlin needs to run initialization code between the super call and property
assignment.

**Generated code (ES2015 target):**

```js
class PreactLLC extends Component {
    // NO constructor() — Kotlin generates a static factory instead:
    static new_io_peekandpoke_kraft_vdom_preact_PreactLLC_2izx5j_k$(props, context, $box) {
        var $this = createExternalThis(this, Component, [props, context], $box);
        $this.props_1 = props;          // Kotlin property
        $this.component_1 =
    ...
        ;        // Kraft component wrapper
        return $this;
    }
}
```

**Why this broke Preact (originally):** Preact instantiates components with `new Ctor(props, context)`. With
ES2015 classes, this calls the inherited `Component()` constructor (since there's no `constructor()` override),
which sets `this.props` but NOT the Kotlin-specific `this.props_1` or `this.component_1`. Result:
`TypeError: this.props_1 is undefined`.

**Impact:** affected ALL Kraft components — the entire `PreactLLC` bridge was broken under ES2015.

**Attempted fixes that failed:**

- `Impl::class.js` — returns the ES2015 class, but `new` doesn't call the Kotlin factory
- JS bridge function wrapping the static factory by name — broke ES5 backward compatibility (no static factory in
  ES5 mode), and required expensive runtime reflection
  (`Object.getOwnPropertyNames` + `startsWith("new_")`)

**Fix that works:** the plain JS function bridge above. `::Impl` naturally resolves to the right thing in each
mode (constructor in ES5, static factory in ES2015), and `new` with an object return value transparently swaps
the dummy `this` for the properly-initialized instance.

### Blocker 2: Konva `import { default }` syntax error — RESOLVED

The konva addon used `@JsModule("konva")` which generated `import { default } from 'konva'` in ES2015 mode.
Invalid JavaScript — `default` is a reserved keyword and cannot be used as a named import binding.

**Fix:** Konva was unused and was removed from the repo. No migration was needed.

## Possible Alternative Solutions (not needed — kept for reference)

These options were considered before the bridge fix was found. They remain valid approaches for anyone hitting
similar issues in a different codebase.

### Option A: Preact `options` hook

Preact exposes lifecycle hooks via `preact.options`. The `__b` (before diff) or `_diff` hook fires before
component creation. Could intercept and call the Kotlin static factory there.

**Pros:** Localized to the Preact bridge.
**Cons:** Relies on Preact internals (double-underscore APIs). Fragile across Preact versions.

### Option B: Rewrite PreactLLC without Kotlin class inheritance

Create the Preact component class hierarchy entirely in JavaScript via `js()` blocks. Kotlin side manages
data/state only, not extending the external class.

**Pros:** Fully decoupled from Kotlin's class generation strategy.
**Cons:** Significant rewrite. Loses Kotlin type safety at the bridge boundary.

### Option C: Wait for Kotlin compiler improvement

The static-factory-instead-of-constructor pattern is a known Kotlin/JS IR limitation. JetBrains may address it.

**Pros:** Zero effort.
**Cons:** Unknown timeline.

### Option D: Selective ES2015

Apply ES2015 target only to modules without Preact components.

**Pros:** Pragmatic partial migration.
**Cons:** Split configuration.

## Original Approach (reference)

### Centralized Configuration

Add the target setting once in the root `build.gradle.kts` via the existing `subprojects` block, plus the root
project's own `js` block. This centralized approach was not used — individual example app `build.gradle.kts`
files got the setting directly, since the migration is being rolled out per-application rather than globally.

```kotlin
js {
    compilerOptions {
        target.set("es2015")
    }
    // ...
}
```

## Compatibility Notes

- **Published .klib artifacts are unaffected.** `.klib` is IR (intermediate representation). The ES target only
  affects final JS codegen in the consuming project.
- **Browser support:** ES2015 is supported in all modern browsers (Chrome 51+, Firefox 54+, Safari 10+, Edge 15+).
- **`js("...")` raw blocks audited:** All usages are safe (object literals, Array.isArray, dynamic imports,
  Object.getOwnPropertyNames). None rely on ES5 `this`-binding.
- **Bridge is dual-mode:** The `PreactLLC` bridge works under both ES5 and ES2015 targets. No conditional code
  needed.
